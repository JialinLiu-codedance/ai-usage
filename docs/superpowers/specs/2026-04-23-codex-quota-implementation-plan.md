# Codex 额度菜单栏应用实现计划

## 目标

基于已确认的设计文档，实现一个单账号、纯本地、纯 probe 模式的 Codex 菜单栏应用：

1. 固定周期主动发起最小 probe 请求。
2. 从 `x-codex-*` 响应头解析 `5h` 和 `7d` 窗口。
3. 在菜单栏面板中展示两个窗口的剩余百分比。
4. 提供设置页、连接测试、低额度提醒和重置提醒。

本计划以 [2026-04-23-codex-quota-menubar-design.md](/Users/liujialin/project/ai-usage/docs/superpowers/specs/2026-04-23-codex-quota-menubar-design.md) 为准，并参考 [UI.pen](/Users/liujialin/project/ai-usage/UI.pen) 中的单账号面板、错误态面板和设置页稿。

## 范围边界

本次实现明确包含：

1. 单账号配置和展示。
2. 正常态、低额度态、错误态面板。
3. 设置页中的刷新设置和通知设置。
4. 一条可用的认证链路。
5. probe 请求、响应头解析、窗口归一化和本地状态持久化。

本次实现明确不包含：

1. 多账号支持。
2. 多平台支持。
3. session 文件解析。
4. Web 私有 usage endpoint。
5. 历史趋势和统计页。

`UI.pen` 中的“多账号总览面板”仅作为未来扩展参考，不进入本次开发计划。

## UI 对照

本次实现直接映射以下设计稿：

1. `额度总览面板`：正常态，作为默认主面板。
2. `额度总览面板 - 低额度`：当最低剩余百分比低于阈值时切换对应样式。
3. `额度总览面板 - 错误`：当刷新失败但存在旧快照时展示错误提示和陈旧数据。
4. `设置页`：本次只实现与单账号、刷新、通知相关的子集。

### 需要按 spec 收敛的 UI 项

`UI.pen` 中有两个地方需要在实现时按当前 spec 收敛，而不是照稿全做：

1. 设置页中的“已连接账号 / 添加账号 / 删除账号”区域，当前只保留单账号配置与重新授权，不做多账号管理。
2. 邮箱字段如果当前认证链路拿不到，可先显示账号名称或留空占位，不把“邮箱”作为阻塞字段。

## 实施阶段

### 阶段 1：项目骨架与基础状态

目标：建立最小可运行的 Tauri + Rust + 前端状态通路。

任务：

1. 初始化 Tauri 应用骨架与目录结构。
2. 建立 Rust 核心模块：
   - `codex_provider`
   - `quota_service`
   - `scheduler`
   - `settings`
   - `secrets`
   - `notifications`
   - `state_store`
3. 定义共享数据模型：
   - `QuotaWindow`
   - `QuotaSnapshot`
   - `RefreshStatus`
   - `ProviderErrorKind`
   - `AppSettings`
4. 建立前后端桥接命令：
   - `get_current_quota`
   - `refresh_quota`
   - `test_connection`
   - `get_settings`
   - `save_settings`
5. 建立前端状态容器，确保 UI 不直接发网络请求。

完成标准：

1. 应用能启动。
2. 前端能读取后端状态。
3. 可以用 mock 数据渲染正常态面板和设置页。

### 阶段 2：Codex provider 与 probe 链路

目标：打通真实 probe 请求与响应头解析。

任务：

1. 定义 provider 输入配置：
   - 认证信息
   - Base URL
   - 超时
2. 实现最小 probe 请求构造器。
3. 实现 HTTP 客户端封装：
   - 请求超时
   - 请求头注入
   - 早关闭响应体
4. 实现响应头解析：
   - `x-codex-primary-*`
   - `x-codex-secondary-*`
5. 实现窗口归一化逻辑：
   - primary/secondary 映射到 `5h` / `7d`
   - `remaining_percent = 100 - used_percent`
   - `reset_after_seconds -> reset_at`
   - 过期窗口归零
6. 实现结构化错误映射：
   - 认证失败
   - 网络失败
   - 超时
   - 缺少响应头
   - 非法响应

完成标准：

1. 使用真实凭证时能拿到 `5h` 和 `7d` 窗口结果。
2. 缺失头、超时、认证失败都能返回明确错误类型。

### 阶段 3：Quota service 与状态存储

目标：把 provider 接到应用状态流转中。

任务：

1. 实现 `quota_service.refresh_quota`：
   - 刷新锁
   - 状态切换为 `refreshing`
   - 调用 provider
   - 成功写入快照
   - 失败保留旧快照
2. 实现 `quota_service.get_current_quota`。
3. 实现 `quota_service.test_connection`，但不污染主状态。
4. 实现 `state_store`：
   - 最新成功快照
   - 当前刷新状态
   - 最近错误摘要
   - 最近刷新时间
5. 实现本地缓存：
   - 最近一次成功快照
   - 最近错误摘要
6. 实现应用启动恢复逻辑：
   - 优先显示上次快照
   - 后台触发首次刷新

完成标准：

1. 刷新成功后 UI 自动更新。
2. 刷新失败后保留旧快照。
3. 应用重启后可先显示本地缓存数据。

### 阶段 4：设置与密钥管理

目标：完成可用的单账号设置链路。

任务：

1. 实现 `AppSettings` 本地持久化：
   - `refresh_interval_minutes`
   - `low_quota_threshold_percent`
   - `notify_on_low_quota`
   - `notify_on_reset`
   - `reset_notify_lead_minutes`
   - `base_url_override`
2. 实现 `secrets` 模块，将敏感认证信息写入 macOS Keychain。
3. 在设置页接通以下功能：
   - 填写认证信息
   - 保存设置
   - 测试连接
   - 手动重新授权
4. 处理设置页结果反馈：
   - 成功
   - 认证失败
   - 网络失败
   - 当前不支持

完成标准：

1. 可以完成一次完整的“输入配置 -> 保存 -> 测试连接 -> 刷新成功”流程。

### 阶段 5：面板 UI 落地

目标：把主面板和错误态按设计稿落地，并绑定真实状态。

任务：

1. 实现正常态面板：
   - 账号名称
   - `5H` / `7D` 剩余百分比
   - 两条进度条
   - 刷新按钮
   - 设置按钮
2. 实现低额度态面板：
   - 当最小剩余百分比低于阈值时切换警告/危险色
3. 实现错误态面板：
   - 错误提示文案
   - “数据来自 X 分钟前”
   - 使用灰态展示最后一次成功快照
4. 实现刷新中状态：
   - 菜单栏图标和按钮反馈
5. 菜单栏主摘要显示规则：
   - 默认显示两个窗口中更低的剩余百分比

完成标准：

1. 正常态、低额度态、错误态都能由真实状态驱动切换。
2. 主面板和 `UI.pen` 的结构基本一致。

### 阶段 6：自动刷新与通知

目标：让应用达到“常驻可用”。

任务：

1. 实现 `scheduler`：
   - 启动首次刷新
   - 固定周期刷新
   - 避免并发任务
2. 实现低额度通知：
   - 以 `5h` / `7d` 最低剩余百分比为准
   - 同一轮低额度状态去重
3. 实现重置提醒：
   - `5h` / `7d` 分别判断
   - 对同一个 `reset_at` 去重
4. 实现通知状态持久化，防止应用重启后重复提醒。

完成标准：

1. 自动刷新按设置周期工作。
2. 低额度和重置提醒均可稳定触发且不会重复骚扰。

### 阶段 7：测试与收尾

目标：补足可交付前的验证。

任务：

1. 单元测试：
   - 响应头解析
   - 窗口映射
   - 百分比计算
   - 过期窗口归零
   - 错误分类
   - 通知去重
2. 集成测试：
   - 正常样本
   - 429 但有 `x-codex-*`
   - 缺失头
   - 非法头值
   - 超时
3. 手工验证：
   - 首次启动
   - 手动刷新
   - 自动刷新
   - 设置保存
   - Keychain 读写
   - 错误态回退
4. 文案和边界检查：
   - 中文错误提示
   - 时间格式
   - 百分比展示

完成标准：

1. 核心链路有自动化测试覆盖。
2. 菜单栏 MVP 可以稳定演示和日常使用。

## 建议实现顺序

按依赖关系，建议严格按照以下顺序推进：

1. 阶段 1：项目骨架与基础状态
2. 阶段 2：Codex provider 与 probe 链路
3. 阶段 3：Quota service 与状态存储
4. 阶段 4：设置与密钥管理
5. 阶段 5：面板 UI 落地
6. 阶段 6：自动刷新与通知
7. 阶段 7：测试与收尾

理由：

1. 没有阶段 1 的状态骨架，后面的 UI 和调度都会返工。
2. 没有阶段 2 的真实 provider，后面的通知和错误态都只能是假数据。
3. 设置页必须晚于状态和 provider，否则只是在堆表单。
4. 通知必须晚于调度和稳定数据源，否则误报概率高。

## 任务拆分建议

如果按开发批次拆，可以分成下面 5 个较稳的交付批次：

1. 批次 A：Rust 数据模型 + mock 状态 + 主面板静态渲染
2. 批次 B：真实 probe 请求 + 响应头解析 + 连接测试
3. 批次 C：状态存储 + 本地缓存 + 设置与 Keychain
4. 批次 D：自动刷新 + 通知 + 错误态与陈旧数据展示
5. 批次 E：测试补齐 + 文案收口 + 交付整理

## 风险与预案

### 风险 1：认证链路不稳定

影响：

1. 无法稳定获取 `x-codex-*` 头。

预案：

1. MVP 只接入一条已验证的认证链路。
2. provider 层保持独立，避免请求细节泄漏到 UI。

### 风险 2：响应头字段变化

影响：

1. 解析逻辑失效。

预案：

1. 解析层对缺失字段做容错。
2. 错误态保留最后一次成功快照。
3. 集成测试保留录制样本，便于回归。

### 风险 3：UI 稿与当前范围不完全一致

影响：

1. 开发中出现“做不做多账号”的歧义。

预案：

1. 明确以单账号 spec 为准。
2. `UI.pen` 中多账号部分不进入本次开发。

## 验收清单

交付前需要逐项确认：

1. 单账号配置可保存，密钥写入 Keychain。
2. 连接测试能区分成功、认证失败、网络失败、当前不支持。
3. 主面板可展示 `5H` / `7D` 剩余百分比。
4. 低额度时 UI 切换到低额度态。
5. 刷新失败时 UI 切换到错误态，并保留旧快照。
6. 自动刷新按设置周期运行。
7. 低额度提醒和重置提醒均可工作。
8. 应用重启后可先显示缓存，再执行首次刷新。
