# Codex 额度菜单栏应用设计文档

## 概述

这个项目是一款基于 Tauri 和 Rust 开发的 macOS 状态栏常驻应用。第一阶段范围刻意收敛，只解决一个问题：监控 Codex 的剩余配额，并通过菜单栏、轻量面板和系统通知为个人用户提供随手可见的额度状态。

产品采用纯本地运行模式，不依赖云端服务。本版只实现一条额度获取链路：

1. 应用按固定周期主动发起最小化 probe 请求。
2. 从响应头解析 `x-codex-*` 字段。
3. 将结果归一化为 `5h` 和 `7d` 两个窗口的剩余百分比与重置时间。

本版不接入本地 session 文件解析，也不依赖私有 usage endpoint。

## 产品目标

第一版需要解决三个明确问题：

1. 用户无需打开浏览器或客户端，就能看到当前 Codex 的 `5h` 和 `7d` 剩余百分比。
2. 当剩余百分比低于阈值时，应用能够及时提醒用户。
3. 整个产品保持轻量，配置和维护成本足够低，适合作为个人工具长期常驻。

## 非目标范围

第一版不包含以下内容：

1. Claude、GLM、MiniMax、Kimi 等多平台支持。
2. 多账号支持。
3. 历史趋势图和长期分析。
4. 基于 session 文件或本地 rollout 文件的额度读取。
5. 依赖 ChatGPT Web 私有 usage endpoint 的额度查询。
6. 导入导出、云同步、团队能力。

## 范围确认

本版明确采用以下产品约束：

1. 只使用纯 probe 模式，固定周期主动请求并解析响应头。
2. 只展示两个窗口配额：`5h` 和 `7d`。
3. UI 只展示“剩余百分比”，不把数据伪装成美元余额或 token 余额。
4. 如果能可靠拿到重置时间，则展示重置时间并支持重置提醒。

## 功能范围

### 1. 状态栏常驻

应用以 macOS 菜单栏工具的形式存在，强调“抬眼即见”的使用体验。

必需行为：

1. 显示常驻的菜单栏图标。
2. 图标或标题区域能体现四种状态：
   - 正常
   - 低额度
   - 刷新中
   - 刷新失败
3. 图标默认显示两个窗口中更低的剩余百分比，作为主状态摘要。
4. 点击图标后展开轻量面板。

### 2. 额度总览面板

菜单栏面板是第一版的主要交互入口，只展示高价值信息，不做复杂后台页面。

必需展示字段：

1. 账号名称
2. `5h` 剩余百分比
3. `7d` 剩余百分比
4. `5h` 重置时间
5. `7d` 重置时间
6. 最近刷新时间
7. 当前刷新状态

必需操作：

1. 手动刷新
2. 打开设置

### 3. 自动刷新

应用需要在后台定时刷新 Codex 配额，无需用户频繁手动触发。

必需行为：

1. 启动后在设置和密钥加载完成后执行首次刷新。
2. 支持固定刷新周期，例如 5 分钟、15 分钟、30 分钟。
3. 防止并发刷新，避免同一时间重复发起 probe 请求。
4. 在刷新开始、成功、失败时及时更新 UI 状态。
5. 刷新失败时保留最近一次成功快照。

### 4. 连接配置

第一版只支持一个 Codex 账号。

设置页需要支持配置用于获取 Codex 配额的认证方式。具体字段在实现阶段只接入最终验证通过的方案，但界面结构允许后续扩展。

设计预留以下输入能力：

1. API Key
2. Session Token
3. Cookie
4. Base URL 覆盖

MVP 实际实现时，只要求接通一条稳定可用的认证链路。

### 5. 连接测试

应用需要提供连接测试能力，让用户在依赖后台刷新之前先确认 probe 配置可用。

必需结果类型：

1. 连接成功
2. 认证失败
3. 网络失败
4. 接口可达但当前未返回可识别的 `x-codex-*` 头

测试结果应当直接在设置页中用可读文本展示，不单独扩展一个复杂诊断页。

### 6. 低额度提醒

用户可以设置一个低额度阈值。当任一窗口的剩余百分比低于该阈值时，应用发送 macOS 原生通知。

必需行为：

1. 支持开启或关闭低额度提醒。
2. 支持设置剩余百分比阈值。
3. 对同一轮低额度状态进行去重，避免短时间内反复打扰。
4. 阈值判断以 `5h` 和 `7d` 中更低的剩余百分比为准。

### 7. 重置提醒

如果 probe 结果可以提供可靠的重置时间，应用应支持重置提醒。

必需行为：

1. 支持开启或关闭重置提醒。
2. 支持在重置前的固定提前量发送通知。
3. `5h` 和 `7d` 窗口分别去重，避免对同一重置点重复通知。
4. 如果某个窗口没有可靠重置时间，则该窗口不参与重置提醒。

## 技术方案

### 核心思路

应用不主动查“余额接口”，而是通过最小化 Codex probe 请求获取当前限额快照。请求目标是拿到响应头，不依赖响应体作为主要数据源。

一次成功 probe 的处理步骤为：

1. 构造最小合法请求。
2. 发送到配置好的 Codex 兼容 endpoint。
3. 读取响应头中的 `x-codex-primary-*` 和 `x-codex-secondary-*` 字段。
4. 将窗口信息归一化为 `5h` 和 `7d`。
5. 计算剩余百分比和重置时间。
6. 写入 `state_store`。

### 响应头范围

本版至少解析以下头：

1. `x-codex-primary-used-percent`
2. `x-codex-primary-window-minutes`
3. `x-codex-primary-reset-after-seconds`
4. `x-codex-secondary-used-percent`
5. `x-codex-secondary-window-minutes`
6. `x-codex-secondary-reset-after-seconds`

如响应中存在以下附加头，可保留为扩展字段，但不作为 MVP 主展示依赖：

1. `x-codex-plan-type`
2. `x-codex-active-limit`
3. `x-codex-credits-balance`
4. `x-codex-credits-has-credits`
5. `x-codex-credits-unlimited`

### 归一化规则

probe 结果统一转换为两个窗口：

1. `5h`
2. `7d`

映射规则：

1. 如果 primary 和 secondary 都带 `window_minutes`，则较小窗口映射为 `5h`，较大窗口映射为 `7d`。
2. 如果只拿到一个 `window_minutes`，则小于等于 360 分钟的窗口视为 `5h`，大于 360 分钟的视为 `7d`。
3. 如果都没有 `window_minutes`，则退回兼容假设，由实现层使用固定映射。
4. `remaining_percent = 100 - used_percent`。
5. `reset_at = fetched_at + reset_after_seconds`。
6. 如果窗口已过期，即 `reset_at <= now`，则该窗口使用率视为 0，剩余百分比视为 100。

## 架构设计

应用采用 `薄前端壳层 + Rust 服务核心` 的结构。业务逻辑尽量放在 Rust 侧，避免前端页面直接承载请求、调度与状态判断。

### 前端职责

Tauri 前端负责：

1. 菜单栏面板渲染
2. 设置页渲染
3. 连接测试触发
4. 手动刷新触发
5. 当前配额状态展示

前端不应包含 Codex probe 请求拼装与响应头解析逻辑。

### Rust 侧职责

Rust 后端负责：

1. probe 请求构造与执行
2. 响应头解析与窗口归一化
3. 刷新调度
4. 通知触发
5. Keychain 访问
6. 本地状态持久化

## 模块划分

### `tray`

负责：

1. macOS 状态栏图标与菜单
2. 状态展示更新
3. 点击行为与面板打开逻辑

### `quota_service`

作为应用的核心服务，统一提供：

1. `refresh_quota`
2. `get_current_quota`
3. `test_connection`

这个服务负责刷新编排，是唯一允许触发 probe 请求的入口。

### `codex_provider`

负责：

1. 认证信息注入
2. 最小 probe 请求构造
3. HTTP 请求发送
4. 响应头解析
5. 统一额度快照输出

### `scheduler`

负责：

1. 启动时首次刷新
2. 定时周期刷新
3. 保证任意时刻只有一个刷新任务在运行

### `settings`

负责：

1. 刷新频率
2. 阈值设置
3. 通知开关
4. Base URL 覆盖
5. 非敏感配置持久化

### `secrets`

负责将敏感凭证写入和读取 macOS Keychain。

### `notifications`

负责：

1. 低额度通知
2. 重置提醒通知
3. 通知去重

### `state_store`

负责维护应用当前的内存态，包括：

1. 最新成功快照
2. 刷新状态
3. 最近刷新时间
4. 最近错误状态

状态栏和面板 UI 都应订阅这里的状态，而不是自己发起网络请求。

## 数据模型

第一版直接使用窗口配额模型，不使用余额型 `remaining/used/total` 抽象。

### `QuotaWindow`

字段：

1. `used_percent`
2. `remaining_percent`
3. `reset_at`
4. `window_minutes`

### `QuotaSnapshot`

字段：

1. `account_name`
2. `five_hour`
3. `seven_day`
4. `fetched_at`
5. `source`

其中：

1. `five_hour` 对应 `5h` 窗口
2. `seven_day` 对应 `7d` 窗口
3. `source` 固定为 `probe_headers`

### `RefreshStatus`

取值：

1. `idle`
2. `refreshing`
3. `ok`
4. `error`

### `ProviderErrorKind`

建议统一为：

1. `unauthorized`
2. `network`
3. `timeout`
4. `unsupported`
5. `invalid_response`
6. `missing_headers`
7. `unknown`

### `AppSettings`

字段：

1. `refresh_interval_minutes`
2. `low_quota_threshold_percent`
3. `notify_on_low_quota`
4. `notify_on_reset`
5. `reset_notify_lead_minutes`
6. `base_url_override`

## 刷新链路

应用遵循单一刷新链路：

1. 应用启动。
2. 从本地加载设置。
3. 从 Keychain 读取密钥。
4. `quota_service` 执行首次刷新。
5. `state_store` 状态切换为 `refreshing`。
6. `codex_provider` 发起 probe 请求并解析响应头。
7. 成功时写入最新快照。
8. 失败时保留最近一次成功快照，并写入错误状态。
9. 状态栏和面板 UI 根据状态重新渲染。
10. `scheduler` 触发后续周期刷新。
11. 通知规则基于最新快照进行判断。

这个流程可以避免重复请求逻辑，也能减少状态不一致问题。

## probe 请求策略

### 请求目标

probe 的目标是拿到带 `x-codex-*` 的响应头，而不是消费完整响应体。

### 请求约束

1. 使用最小合法请求体，降低额度消耗和延迟。
2. 设置总超时，避免后台任务长时间卡住。
3. 在拿到响应头后即可进入解析流程，响应体可尽快丢弃或关闭。
4. 连接测试与正式刷新应共用同一套请求拼装逻辑。

### 请求结果判定

1. 如果能解析出有效 `x-codex-*` 头，视为成功。
2. 如果返回认证错误，视为认证失败。
3. 如果请求超时或网络不可达，视为网络失败。
4. 如果接口返回成功但没有可识别的 `x-codex-*`，视为当前不支持。

## 连接测试策略

`test_connection` 只负责验证配置是否可用，不写入主状态。

返回四类结果：

1. 连接成功
2. 认证失败
3. 网络失败
4. 接口可达但当前不支持

设置页应直接显示用户可理解的文本，不暴露底层错误枚举。

## 通知策略

### 低额度提醒

规则：

1. 使用 `5h` 和 `7d` 两个窗口中更低的 `remaining_percent` 作为判定值。
2. 当判定值首次低于阈值时发送通知。
3. 当判定值持续处于阈值以下时不重复通知。
4. 当判定值恢复到阈值以上后，解除去重锁。

### 重置提醒

规则：

1. `5h` 和 `7d` 窗口独立判断。
2. 当距离某窗口的 `reset_at` 小于设定提前量时发送通知。
3. 对同一个 `reset_at` 只提醒一次。
4. 如果没有可靠 `reset_at`，则跳过该窗口。

## 存储策略

敏感数据必须存储在 macOS Keychain 中。非敏感配置和最近一次额度快照可以存储在本地。

对于 MVP，存储方案应优先考虑简单可用：

1. 凭证使用 Keychain 存储。
2. 普通设置使用轻量本地持久化。
3. 缓存最近一次成功快照，以便应用启动后快速显示上次状态。
4. 缓存最近一次错误摘要与通知去重状态。

## 错误处理

第一版虽然不做详细日志系统，但仍然需要清晰的失败行为。

必需行为：

1. 刷新失败时保留最近一次成功快照。
2. 将应用状态标记为失败。
3. 在 UI 中展示简洁可读的失败状态。
4. 允许用户从菜单栏面板手动重试。
5. 不因一次 probe 失败而清空已有窗口配额展示。

## 测试策略

MVP 至少覆盖三层验证：

1. 单元测试：响应头解析、窗口归一化、过期窗口归零、阈值判断、通知去重。
2. 集成测试：基于录制样本验证 `codex_provider` 的 probe 解析逻辑，包括正常响应、429 且带 `x-codex-*`、缺失头、非法值等情况。
3. macOS 手工验证：状态栏行为、Keychain 存储、通知触发、启动后自动刷新、刷新失败保留旧快照。

## MVP 成功标准

如果满足以下条件，就可以认为第一版达到了可用标准：

1. 用户能够在本地成功配置一个 Codex 账号。
2. 应用能够稳定发起 probe 并展示 `5h` 和 `7d` 剩余百分比。
3. 菜单栏状态始终能反映当前应用健康状态。
4. 低额度提醒可以工作且不会重复骚扰。
5. 连接测试能够明确指出基础配置问题。

## MVP 后续迭代顺序

当前版本稳定后，后续应按以下顺序扩展：

1. 先加固 probe 兼容性与错误恢复。
2. 再补充 plan type、credits 等附加信息展示。
3. 再增加 session 文件读取作为备用数据源。
4. 再扩展到多账号支持。
5. 最后再考虑多平台总览和历史统计。
